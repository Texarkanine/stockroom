# Brainstorm ‚Äî An On-Path `stockroom` CLI, Installed by Onboarding

Design sketch (2026-07-07): `sr-initialize` (the onboarding skill, decision D9) additionally binds an on-path **`stockroom` command** that owns all invocation plumbing, so every skill's engine contract collapses to "run `stockroom <subcommand> ‚Ä¶`". Recorded for later action ‚Äî nothing here has been applied.

Feeds: Phase 3 ‚Äî Onboarding, CLI, and Scheduling (`sr-initialize`) in `planning/roadmap.md` (this document was adopted into that phase by the 2026-07-08 roadmap resequencing, `memory-bank/active/creative/creative-roadmap-resequencing.md`); the wrapper-skill trimming pass in `skill-litter-audit.md`; the Phase-5 per-harness invocation verification.

## Problem

Every skill that shells into the engine must today reproduce a four-part incantation: resolve `$APP_DIR` (plugin-root env var + `find -L` fallback), set `PYTHONPATH="$APP_DIR/src"`, and pass `--no-sync --no-config` to `uv run`. Each part is load-bearing; omitting any one fails the call or (worse, for `--no-sync`) silently strips torch. The incantation is copy-pasted prose across `sr-query`, `sr-semantic`, soon `sr-search`, `systemPatterns.md`, and the README ‚Äî and it already drifted into being wrong in N places once (caught and fixed in the m4 sub-run). The m4/m5 reflections both flagged a shared launcher as the obvious consolidation; this document is that idea taken to its logical end.

The insight that makes it an *onboarding* concern: the initializer is the one component that already must understand how the system works (torch provisioning, wheel selection, scheduling, first ingest/embed ‚Äî Phase 3's existing scope). Let it also be the one place that understands how the system is *invoked*. Then skills need to know exactly one thing: `stockroom ‚Ä¶` is on PATH, and `--help` explains the rest.

## Shape

### The shim

`sr-initialize` writes an executable shim to `~/.local/bin/stockroom` (checking that the directory is on PATH and saying so if not). The shim does environment plumbing only, then hands off:

```sh
#!/bin/sh
# generated by sr-initialize ‚Äî re-run sr-initialize to regenerate
APP_DIR="<baked-at-install-time>"
[ -f "$APP_DIR/pyproject.toml" ] || APP_DIR="$(re-resolve via the find -L fallback)"
PYTHONPATH="$APP_DIR/src" exec uv run --project "$APP_DIR" --no-sync --no-config \
  python -m stockroom "$@"
```

Two properties are the point:

- **The torch contract becomes structural.** `--no-sync` / `--no-config` are hard-coded in one generated file; no agent or user can forget them, and no skill needs to explain them.
- **Bake-then-verify survives plugin updates.** Harness plugin caches are versioned per release, so a baked `APP_DIR` goes stale on update. The shim verifies the baked path and falls back to runtime re-resolution ‚Äî fast when valid, self-healing when the plugin moves. (Re-running `sr-initialize` also regenerates it; the shim is idempotent to reinstall.)
  - TODO: figure out how updates won't result in an on-path `stockroom` pointing at an old or invalid set of python

### The dispatcher

`python -m stockroom` (a new, small `stockroom/__main__.py`) dispatches subcommands to the existing modules: `stockroom query ‚Ä¶` ‚Üí `stockroom.query`, likewise `semantic`, `ingest`, `embed`, `migrate`. Dispatch logic lives in tested Python (the TDD rule binds it; the shim stays too dumb to need tests). `stockroom --help` lists the subcommands; `stockroom <sub> --help` forwards to each module's existing argparse ‚Äî the "just a CLI with `--help`" property the skills will lean on.

`[project.scripts]` was considered and rejected: console-script entry points require the package to be installed, and the engine is deliberately run-in-place (`[tool.uv] package = false`, no build step). The generated shim is the packaging-honest equivalent.

### The skill contract after this lands

- If `command -v stockroom` succeeds: run `stockroom <subcommand> ‚Ä¶`. Everything else about invocation disappears from skill prose ‚Äî no `$APP_DIR`, no `PYTHONPATH`, no uv flags, no rationale.
- If it fails: tell the user to run `sr-initialize`. **No fallback incantation in skills** ‚Äî a fallback would reintroduce exactly the litter and drift this removes (see `skill-litter-audit.md`).
- Understanding how the system works lives in the initializer and the shared reference doc, not in the wrapper skills.

A pleasant side effect for Phase 5: the *engine* invocation form stops being per-harness at all (PATH is PATH in both Cursor and Claude Code), shrinking the empirical verification surface to the skill-invocation forms (`/sr-*` vs `<plugin>:<skill>`).

## Open questions for the Phase-3 sub-run

- **Resolution order when both harness caches exist** (Cursor and Claude both have the plugin installed): pick deterministically ‚Äî e.g. prefer the path the generating harness provided at install time, fall back to newest `pyproject.toml` mtime ‚Äî and record the choice in the shim comment.
- **Shim template location**: ship it as a file in the engine (`skills/sr-search/scripts/stockroom-shim.sh` or similar) that `sr-initialize` copies and parameterizes, so the template is reviewable and REUSE-covered rather than a heredoc buried in skill prose.
- **`uv` missing at shim runtime**: the shim should fail with one clear line ("uv not found ‚Äî see sr-initialize prerequisites"), not a bare shell error.
- **Dev-repo ergonomics**: developers working in the repo want the same one-word entry; a `make shim` (or `sr-initialize --dev` pointing at the checkout) keeps dev and installed behavior identical. Relatedly, decide whether the repo `Makefile` targets start delegating to the dispatcher.
  - TODO: perhaps a `stockroom-dev` shim that explicitly goes to the active repo, for convenient iteration? But dev-installed skills would still use `stockroom` since they're also tested as-written. Do it (testing cli is useful for local dev?) or not - need to investigate and weigh benefit vs "gotcha" of thinking a local SKILL tested the local CODE. Currently local skills DO test local code.
- **Skill naming**: the roadmap and D9 name the skill `sr-initialize`; keep that (the operator's "sr-init or something" resolves to the already-decided name).
